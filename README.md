# react-debounce-search-sample

A sample React application demonstrating a **reusable debounced text field** component, Redux async state management with `AbortSignal`-based request cancellation, and stale response discarding.

> **Note:** This codebase was generated by [Claude Code](https://claude.ai/claude-code) (Anthropic).

## Overview

This sample demonstrates two key patterns:

1. **Generic debounced input** — `SearchInput` is a reusable MUI `TextField` wrapper that debounces `onChange` callbacks. It has no knowledge of Redux or any specific business logic.
2. **Request cancellation via AbortSignal** — `App.tsx` uses `thunkPromise.abort()` to cancel the previous in-flight request before dispatching a new one. The fake API respects `AbortSignal`, so the timer is cleared immediately and RTK dispatches a `rejected` action with `meta.aborted: true`.

### Key behaviors

| Behavior | Implementation |
|---|---|
| Debounced input (500 ms, configurable) | `SearchInput` — `setTimeout` + `clearTimeout` via `useRef` |
| Enter key flushes debounce immediately | `SearchInput` — `flushDebounce()` reads latest value from ref |
| Previous request cancellation | `App.tsx` — `pendingRequest.abort()` before dispatching new thunk |
| Aborted request counter | `searchSlice` — `abortedCount` incremented on `rejected` with `meta.aborted` |

## How It Works

### SearchInput (generic debounced TextField)

```
onChange  →  setValue + update ref  →  clearTimeout  →  setTimeout(onChangeRef.current, 500ms)
onKeyDown(Enter)  →  flushDebounce()  →  clearTimeout + onChangeRef.current(latestValueRef.current)
```

- `onChange` prop is stored in a ref to avoid stale closures in the debounce timer
- `latestValueRef` ensures flush always uses the most recent input value, regardless of React's batched state updates
- Accepts all MUI `TextFieldProps` via rest spread, and an optional `delay` prop to customize the debounce interval

### Request cancellation (App.tsx + Redux)

```
User types "a"   → (after debounce) dispatch(fetchSearchResults("a"))
User types "ab"  → pendingRequest.abort()  →  rejected(meta.aborted)  →  abortedCount++
                   dispatch(fetchSearchResults("ab"))  →  fulfilled  →  results updated
```

The fake API (`searchThunks.ts`) respects `AbortSignal`: on abort, the in-flight timer is cleared immediately and the promise rejects with `AbortError`. This guarantees no stale `fulfilled` action ever reaches the reducer.
