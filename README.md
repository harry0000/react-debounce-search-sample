# react-debounce-search-sample

A sample React application demonstrating debounced text input with immediate-on-Enter search, Redux async state management, and stale response discarding ‚Äî without relying on AbortController inside the API layer.

> **Note:** This codebase was generated by [Claude Code](https://claude.ai/claude-code) (Anthropic).

## Overview

This sample covers a real-world pattern where the API is a wrapped library that **cannot be aborted**. Instead of attempting to cancel in-flight requests, the app lets every API call run to completion and discards stale responses in the Redux reducer by comparing `requestId` values.

### Key behaviors

| Behavior | Implementation |
|---|---|
| Debounced input (500 ms) | `setTimeout` + `clearTimeout` via `useRef` |
| Enter key triggers search immediately | `onKeyDown` bypasses the debounce timer |
| Stale response detection | `latestRequestId` tracked in Redux state |
| Previous API call "cancellation" | Stale `fulfilled` actions are discarded in the reducer |
| Discarded response counter | `abortedCount` incremented on stale `fulfilled` |

## How It Works

### Debounce + Enter

```
onChange  ‚Üí  clearTimeout  ‚Üí  setTimeout(triggerSearch, 500ms)
onKeyDown(Enter)  ‚Üí  clearTimeout  ‚Üí  triggerSearch() immediately
```

### Stale Response Discarding

```
dispatch(fetchSearchResults("a"))   ‚Üí pending  ‚Üí latestRequestId = "req-1"
dispatch(fetchSearchResults("ab"))  ‚Üí pending  ‚Üí latestRequestId = "req-2"

fulfilled("req-1") arrives  ‚Üí  req-1 ‚âÝ latestRequestId  ‚Üí  abortedCount++  (discarded)
fulfilled("req-2") arrives  ‚Üí  req-2 = latestRequestId  ‚Üí  results updated
```

This pattern is necessary when the API is a wrapped library that does not expose an `AbortSignal` interface. Calling `.abort()` on the RTK thunk promise would cause the `rejected` action to fire immediately, but the underlying API would still run to completion ‚Äî making `meta.aborted` an unreliable signal for counting discards.
